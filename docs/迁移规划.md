# 景区游船项目 — 云开发 → 自有服务器迁移规划

> 创建于 2026-05-26，状态：**待定**（先完成客户新需求，再决定是否迁移）

## 背景

当前全部走微信云开发（云函数 + 云数据库 + 静态托管）。admin-web 后台是 2026-02 快速交付时的临时方案（单 HTML 文件 2555 行，CloudBase JS SDK 直连云数据库），随功能增加难以维护。迁移到自有服务器后可以用正式的前后端分离架构重建后台。

## 迁移范围

### 1. 后端 API（替代 39 个云函数）

- 用 Node.js（Fastify）统一起一个后端服务
- 云函数逻辑大部分很薄（CRUD + 微信支付），迁移工作量不大
- 需要实现：JWT 鉴权、微信登录（code 换 openid）、中间件等

### 2. 数据库（替代云数据库）

云数据库是 MongoDB 兼容的文档型，迁移两条路：

| 方案 | 优势 | 劣势 |
|------|------|------|
| MySQL/PostgreSQL | 关系型结构清晰，适合订单/财务数据 | 需要重新建表、写迁移脚本 |
| MongoDB | 几乎平迁，字段结构基本不改 | 需要额外装 MongoDB |

### 3. 小程序端

- `wx.cloud.callFunction` → `wx.request` 调自有 API
- 登录流程：小程序拿 code → 后端换 openid/session_key
- 改动面不大，但所有云函数调用点都要改

### 4. 微信支付

- 回调地址从云函数 HTTP 触发改为自有服务器公网 URL
- 支付签名验签逻辑不变

### 5. admin-web 重建（迁移最大收益）

- Vue3 + Vite 正式项目，组件化、路由、权限控制
- 后端有正式 API 后，后台直接调接口

## 关键风险

- **数据迁移**：线上已有真实订单和用户数据，需写迁移脚本，不能丢
- **微信支付回调**：切换期间新旧回调都要能收到，否则丢支付通知
- **小程序审核**：改了请求域名需在小程序后台配服务器域名白名单，重新提审
- **过渡期并行**：可能需要一段时间新旧并行，确保稳定后再切

## 可选策略

1. **全量迁移**：后端 + 数据库 + 小程序 + 后台一起迁，一步到位但风险大
2. **只迁后台**：先搭自有后端 API + 新 admin-web，小程序端暂时不动（后端同时支持云函数和 API）
3. **继续云开发**：如果客户需求不复杂，维持现状也行

## 决策时机
